# GPT Image 2 — 카탈로그 아이템별 프롬프트 스펙 (진행 중)

## 0. MISSION

epicstage-kv-studio의 **Step 4 바리에이션 생성**은 94종(KV 3 + MASTER_CATALOG)의 제작물을 GPT Image 2로 생성한다. 1차 리팩토링(→ `prompts-openai.ts` / `production-image-openai.ts`)은 **일반 프롬프트 구조**만 GPT Image 2 양식으로 바꿨다. 하지만 카탈로그의 각 아이템은 **성격이 달라** (원거리 대형 LED, 텍스트 밀도 높은 공식 문서, 중앙 비움 대지, 시리즈 카드뉴스 등) 단일 템플릿으로는 부족하다.

이 작업은 **84개 아이템(굿즈 9종 제거 후)을 하나씩 돌면서** 각 아이템이 프롬프트 엔진에 어떤 커스터마이징을 요구하는지 결정해 기록한다. 84개 전부 결정된 뒤 **한 번에 코드(타입·카탈로그·프롬프트 빌더)에 반영**한다.

**스킵 대상**: KV 3종은 Step 3(`master-kv-openai.ts`)에서 이미 다룸 → 이 리뷰 대상 아님.

---

## 1. 대화 규칙 (세션 재시작 후에도 반드시 지킬 것)

1. **5개씩 배치**로 제안한다. (2026-04-24 변경: 1개씩 → 5개씩, 속도 우선)
2. **제안 포맷** — 배치지만 **아이템당 충분히 풀어서 설명** (약어·플래그명만 나열 금지):
   - **아이템이 뭐고 어떻게 쓰이는지** (1~2문장으로 현장 상황 묘사)
   - **텍스트 구조 제안**: headline만? subtext까지? 없음? 특수 필드? 왜 그렇게 정했는지
   - **특수 플래그 제안**: 왜 필요한지 사람 말로 설명 (플래그명 + 실제 효과)
   - **판단 포인트**: 사용자가 O/X/수정을 고르기 위해 봐야 할 핵심 질문
3. 사용자 답은 배치 단위로 `O` (전체 수락) / 개별 `N번 X` 또는 `N번 수정: …`. 답 오면 **즉시** 이 파일을 업데이트하고 다음 배치로.
4. **코드는 아직 수정하지 않는다** — 84개 전부 결정된 뒤 한 번에.
5. 새 플래그/필드가 필요하면 **섹션 4 "전역 개념"에 먼저 추가**한 후 아이템 결정에 사용.
6. 사용자가 "빼자" 하면 `제거` 열에 O, 간단한 이유만 기록.
7. **짧게 응답**한다. 설명 과잉 금지. 결정 중심.

## 4. 전역 개념 (새 플래그/필드) — 아이템 리뷰하며 누적

| 개념 | 타입 | 용도 | 프롬프트 반영 |
|---|---|---|---|
| `customSize` | `{widthMm, heightMm}?` | 아크릴픽·마이크택처럼 실물 크기 특수 | mm → px 변환해 `size` 파라미터로 |
| `physicalSizeMm` | `{widthMm, heightMm}` (카탈로그 고정값) | 한국 인쇄 실측 규격이 명확한 아이템(명찰·명패 등). **비율로 근사 X, mm 고정**. 생성 단계에서 mm → px(300dpi 등) 변환해 `size` 파라미터로. 프롬프트에도 "exact physical dimensions: {W}mm × {H}mm" 명시. | "Designed for print at exactly {W}mm × {H}mm ({W/H ratio}) — maintain that aspect precisely." |
| `bulkCsvOverlay` | `boolean` (카탈로그 필드) | **LLM 생성은 템플릿 1장만**. 이후 사용자가 업로드한 CSV(예: 직함/이름/소속 컬럼)를 읽어 템플릿의 safeZone 영역에 **코드로 텍스트 오버레이**(Canvas/PIL 등) 해서 N명분 PNG를 일괄 출력. 추가 LLM 호출 0회 = 비용 0. **구현은 섹션 8-2 "별도 관리"** 로 분리 (Step 4 본체와 별도 파이프라인). | (프롬프트에는 영향 없음 — 템플릿 프롬프트는 safeZoneRequired 아이템과 동일. UI·후처리 레이어 플래그) |
| `displayDistance` | `"near" \| "mid" \| "far"` | 관람 거리. `far`는 원거리 가독성 우선(대형 타이포, 고대비, 디테일 단순화) | "Viewed from far distance — oversized typography, high contrast, simplified details." |
| `safeZone` | `{ x: number, y: number, width: number, height: number }[]` **(픽셀 단위, 모든 safeZone은 퍼센트가 아닌 픽셀로 표현)** | **사용자 정의**: "대지(배경)만 만들고, 그 영역엔 다른 화면(PPT·실시간 영상·공지 등)이 얹히므로 비워둔다." Step 4 UI에 **safeZone 라디오 버튼(ON/OFF)**. ON이면 **픽셀 좌표 bounding box** 입력 (x, y, width, height). 이미지 렌더 해상도 기준 절대 픽셀. 오버레이 후처리(`bulkCsvOverlay`)와 단위 일치 → 좌표 변환 로스 없음. 일반 아이템은 기본 OFF. | (라디오 ON일 때만 반영) "Keep a rectangular region at (x={x}px, y={y}px, width={width}px, height={height}px) completely empty — reserved for overlay content." |
| `safeZoneRequired` | `boolean` (카탈로그 필드) | 일부 아이템(예: PPT 장표, 화상회의 배경)은 **safeZone이 본질**이므로 토글이 **기본 ON + 끌 수 없음**. 이 플래그가 `true`면 UI에서 safeZone **픽셀 좌표** 입력이 필수. | 해당 아이템 생성 시 safeZone 값 입력 강제. 값 없으면 합리적 기본값(예: 1920×1080에서 중앙 1600×900 bounding box) 제시. |

## 8. 별도 관리 항목 — 카탈로그 본체와 분리해서 구현 검토

카탈로그 스펙 결정 과정에서 "GPT Image 2 Step 4 프롬프트 튜닝 범위를 벗어나는" 과제들이 발견됨. 이들은 별도 트랙으로 관리해 **Step 4 완료 후 또는 병행으로 검토**한다.

### 8-1. #41 브로셔 — 멀티 필드 UI 가능성 검토

**상황**: A4 2단 접지 브로셔는 커버 한 면만 이미지 생성하면 반쪽. 실용성을 가지려면 **행사 타임라인·세션 리스트·연사 소개** 등 구조화된 정보가 내부 페이지에 들어가야 함.

**검토 과제**:
- **(a) UI로 이 정보들을 받을 수 있는가**: `customFields`(현행)가 "증서" 수준의 2~3개 필드를 감당하는데, 브로셔는 **가변 길이 타임라인/세션 N개**가 필요 → 동적 배열 입력 UI 필요. Step 4 카탈로그 UI 패턴이 감당 가능한지 판단.
- **(b) GPT Image 2가 타임라인·긴 목록 텍스트를 제대로 렌더하는가**: 텍스트 밀도 높은 공식문서 스타일. 일반 prompt로는 한계. 텍스트 오버레이 하이브리드가 필요할 수도.
- **(c) 대안 — 템플릿 배경만 GPT Image 2로 + 실제 타임라인/세션은 코드로 오버레이** (명찰 `bulkCsvOverlay` 패턴의 브로셔 버전).
- **결론 후보**: 
  - Option A: 브로셔를 카탈로그에서 제거 (Full 티어 디자이너 직접 처리)
  - Option B: 하이브리드 — 표지·배경은 Step 4, 내지는 별도 오버레이 파이프라인
  - Option C: 미래 기능으로 보류, 지금은 커버 전용 엔트리로만 (38·39·40 제거 논리와 충돌)

**담당**: 프로덕트 + AI + 엔지니어링. **우선순위**: Step 4 본체 완료 후.

### 8-2. 명찰·명패 `bulkCsvOverlay` — 대량 제작 파이프라인

**상황**: 명찰(#42~44)·명패(#45+46)는 참가자 100명 이상 행사에서 쓰는 일이 많음. 한 장씩 생성하면 의미 없고, 이름·직함 데이터를 **CSV로 받아 일괄 생성**해야 실용적.

**핵심 아이디어**: LLM 호출은 **템플릿 1장**만. 이후는 **코드로 텍스트 오버레이**.

```
[1] 사용자: 카탈로그에서 "명찰 (가로)" 선택 + safeZone 영역 설정(이름이 들어갈 자리)
[2] GPT Image 2: 배경 템플릿 1장 생성 (PNG)
[3] 사용자: CSV 업로드 (컬럼: 직함, 이름, 소속 등)
[4] 서버: 각 행마다 템플릿 복제 → safeZone 좌표에 한국 폰트로 텍스트 렌더 (Canvas/Sharp/Pillow)
[5] 결과: N명분 PNG + 인쇄용 PDF 또는 ZIP
```

**구현 검토 포인트**:
- **렌더 기술**: 서버 사이드 — Node.js `@napi-rs/canvas` / `sharp` + 커스텀 폰트, 또는 Python PIL. 프런트 단독은 대량 처리엔 불안정.
- **한글 폰트**: CI 기반 폰트 필수 (Pretendard, Noto Sans KR 등). 폰트 내장 필요.
- **safeZone 좌표 스키마**: 지금 `safeZone`은 "side + fraction"만 있음 → 오버레이용으로 **픽셀 단위 bounding box**(x, y, w, h, 정렬, 폰트 크기)로 확장 필요. 카탈로그 엔트리에 기본값 박아두고, 사용자가 템플릿 생성 후 시각적으로 조정하는 UI.
- **CSV 스키마**: 아이템별로 다름. 명찰은 `name, title, org`, 명패는 `name, title`. 초대장(#47)까지 확장하면 `name, org, table_no` 등. **아이템별 CSV 스키마 정의**를 카탈로그 필드로.
- **미리보기**: CSV 업로드 후 첫 1~3행 샘플만 즉시 렌더해 확인 후 전체 일괄 처리.
- **출력**: 개별 PNG ZIP / 인쇄용 PDF(A4 페이지당 N장 배치) / 디지털 단일 PNG 중 선택.

**결론**: 이건 **Step 4와 분리된 별도 제품 기능** ("Bulk Nametag Generator"). Step 4는 템플릿 1장만 책임지고, 대량 생성 파이프라인은 별도 API/UI로 구축.

**담당**: 엔지니어링(렌더 파이프라인) + 프로덕트(UI/UX) + AI(safeZone 자동 탐지 아이디어 있는 경우).
**우선순위**: Step 4 본체 완료 직후. 예상 규모: 1~2주 작업.

### 8-3. `physicalSizeMm` vs `customSize` 정리

현재 두 필드가 비슷해서 혼란 여지:
- `customSize`: **사용자 지정** mm (아크릴픽·마이크택·안내 POP처럼 현장 맞춤)
- `physicalSizeMm`: **카탈로그 고정** mm (명찰·명패처럼 인쇄 표준 규격)

→ 구현 단계에서 **한 필드로 통합할지(유니언 타입), 명확히 분리할지** 결정 필요. 일단 스펙은 분리 상태로 기록.

### 8-4. 명찰·명패 실측 mm 규격 DB 확보

`physicalSizeMm` 필드에 들어갈 한국 표준 규격 실측값 확보 필요:
- [ ] #42 카드형 명찰 (ISO 카드 표준 86×54mm로 추정, 확인 필요)
- [ ] #43 가로 명찰 (현장 실측값 불명)
- [ ] #44 세로 명찰 (현장 실측값 불명)
- [ ] #45 좌석 명패 실측
- [ ] #46 심사위원 명패 실측

**담당**: 리서치 / 프로덕트. **우선순위**: Step 4 카탈로그 항목 UI 반영 전까지.

### 8-5. #66 SNS 카드뉴스 (다중 슬라이드) — 기획부터 다시

**상황**: 인스타 카드뉴스는 **N장(보통 3~10장) 슬라이드 시퀀스**다. 표지 + 연사 소개 + 프로그램 + 후기 등 순차 스토리텔링. 단일 이미지 카탈로그의 "H/S 한 쌍"으로는 표현 불가능.

**기존 시스템 한계**:
- `production-grid.tsx`는 **단일 생성 루프** — 아이템당 1장만 생성. N장 시리즈 API/UI 없음.
- 프롬프트 구조(`ProductionOpenAiInput`)도 슬라이드 배열 필드 없음.
- 슬라이드 간 **시각적 일관성**(컬러·타이포·레이아웃 락) 유지 메커니즘 부재. GPT Image 2 reference 체이닝 전략 미정.

**차후 프로젝트로 분리하는 이유**: Step 4 본체 리팩토링(단일 템플릿 프롬프트 튜닝)과는 **완전히 다른 제품 기능**. 카탈로그 플래그 몇 개로 해결될 문제가 아니라 UI·API·일관성 엔진·레퍼런스 관리를 전부 새로 설계해야 함.

**기획 재시작 시 검토 과제**:
- **(a) 슬라이드 구성 UI**: 동적 N장(추가/삭제/순서 변경), 슬라이드별 H/S 또는 롤(표지/본문/CTA) 지정
- **(b) 일관성 엔진**: 첫 슬라이드 = KV 기반 마스터 → 이후 슬라이드는 첫 결과를 reference로 고정 체이닝. 또는 `design-system-openai.ts` 출력을 강하게 락
- **(c) 비용/레이트 리밋**: 슬라이드당 GPT Image 2 1회. 10장이면 10배 비용 — 예상 비용 표시 + 부분 실패 재시도 UX
- **(d) 출력 포맷**: 개별 PNG ZIP / 인스타 업로드 포맷 / 프리뷰 캐러셀
- **(e) 카드뉴스 템플릿 프리셋**: "연사 소개 5장", "행사 리뷰 8장" 등 사전 구성 시나리오

**결론**: Step 4 카탈로그에서 **#66은 보류**(섹션 6). 실제 구현은 별도 기획/개발 프로젝트로.

**담당**: 프로덕트 기획부터 재시작 → 디자인 → 엔지니어링. **우선순위**: Step 4 본체 완료 후 별도 논의.

### 8-6. #75 EDM 초청장 — 아예 리디자인

**상황**: 이메일 뉴스레터에 삽입되는 디지털 초청장. 현재 H/S 구조로는 부족하고, **이메일 매체 특유 제약**이 카탈로그의 일반 프롬프트 빌더 범위를 벗어남.

**재설계해야 하는 이유**:
- **이메일 클라이언트 호환성**: Gmail·Outlook·네이버 메일 등에서 이미지 로딩 차단·렌더 깨짐·크기 제한. 단일 이미지 JPEG/PNG 최적화(파일 크기, DPI) 필요.
- **CTA 버튼**: "참가 신청" 같은 버튼은 **이미지에 박히면 클릭 불가**. HTML `<a>`로 처리하거나, 버튼 영역을 safeZone으로 비워 링크 오버레이로 처리.
- **구조화 필드**: 일시·장소·주최·후원·CTA 문구·링크 URL 등. 단순 `customTextUI`로는 부족, `customFields` 세트 재정의 필요.
- **다크 모드 대응**: 일부 메일 클라이언트가 이미지 색상 자동 반전 → 다크/라이트 양쪽에서 가독성 확보 디자인 필요.
- **개별화(merge tag)**: 수신자 이름 `{{name}}` 삽입 가능한 구조 — 브로셔(#41)·명찰의 bulkCsvOverlay와 비슷한 개별화 엔진 공유 여지.

**기획 재시작 시 검토 과제**:
- **(a) 매체 결정**: 단일 이미지 vs HTML+이미지 조합. HTML이면 이미지 생성 범위가 "배경+비주얼만"으로 축소됨.
- **(b) CTA 오버레이 아키텍처**: 이미지엔 버튼 자리 safeZone만, 실제 CTA는 HTML 오버레이. 카탈로그 `safeZone` 픽셀 스키마와 연동.
- **(c) 개별화 엔진 공유**: 명찰 `bulkCsvOverlay` 파이프라인(섹션 8-2) 확장해 "수신자 리스트 CSV → 개별 이미지 N개"로 처리 가능.
- **(d) 이메일 템플릿 프리셋**: "세미나 초청", "파티 초대", "웨비나 등록" 등 시나리오별.

**결론**: Step 4 카탈로그에서 **#75는 보류**. 별도 "EDM 제너레이터" 트랙으로 이관.

**담당**: 프로덕트(이메일 UX·컴플라이언스) + 디자인(다크모드·크로스 클라이언트) + 엔지니어링(HTML+이미지 파이프라인). **우선순위**: 카드뉴스(#66)와 함께 Step 4 본체 완료 후.
